What Is Terraform?

Before writing a single line of code, understand the WHY — the mental model that makes everything else click.

🏗️

Infrastructure as Code (IaC)

Terraform lets you describe your entire cloud infrastructure in text files — VMs, networks, databases, IAM — and then creates, modifies, and deletes that infrastructure automatically. No clicking in consoles. No manual steps. Everything is reproducible, version-controlled, and reviewable.

IaC = "I Always Commit"
Infrastructure · Always in code · Commit to Git

Level 1 · Core Commands

The Sacred Commands

A chef has 3 sacred rituals before every service: mise en place (set up the kitchen), taste test (check the recipe), plate and serve (execute). Terraform has the same ritual: init sets up the kitchen, plan is the taste test, apply plates the dish. Skip any step and serve disasters.

Mnemonic: "I Plan to Apply" → Init → Plan → Apply. Always in this order. Never skip plan. The chef who skips tasting serves disasters.

🏗️
INIT
Download providers
✅
VALIDATE
Check syntax
✨
FMT
Format code
🔍
PLAN
Show changes
🚀
APPLY
Create infra
💣
DESTROY
Tear down
terraform init — Downloads providers into .terraform/, creates .terraform.lock.hcl, connects to the backend. Like npm install. Run once per project and after adding new providers or modules.
terraform validate — Checks syntax and internal consistency without making any API calls. Zero side effects. Run as your first CI check. Catches typos, missing required fields, wrong types.
terraform fmt — Auto-formats .tf files to canonical style (consistent indentation, alignment). Run terraform fmt -check -recursive in CI as a blocking PR gate.
terraform plan — The most important command. Queries cloud APIs, reads state, computes the diff. Shows + create / ~ update / - destroy. Always read this output before applying. Save with -out=tfplan for reproducible applies.
terraform apply — Executes the plan. Creates/updates/destroys real cloud resources. Prompts for confirmation unless -auto-approve. Updates state file after every change. In CI/CD, apply only on merge to main.
terraform destroy — Removes ALL resources managed by this config. Plan first with terraform plan -destroy. Use lifecycle.prevent_destroy = true on databases and state buckets to guard against accidents.
📋 terraform state list

Like reading your inventory ledger
🔎 terraform state show

Level 2 · State

Terraform State — The Memory

📚
STATE = LIBRARIAN'S LEDGER

Why State Exists

Without state, Terraform would have to query every cloud resource on every run — slow and unreliable. State is the cached map of reality. Terraform compares your desired state (code) against known state (file) to compute the minimum diff. It also stores cloud-returned values (IPs, IDs) that you can't know until after creation.

📦Purchase OrdersWhat you ordered (desired)
≠
📋Inventory LedgerWhat's actually on the shelf (state)

Terraform computes the DIFFERENCE between the two → only makes the minimum changes needed

terraform.tfstate — What's Inside
{ "version": 4, // state format version "terraform_version": "1.7.0", "serial": 12, // increments on every apply "lineage": "a3f2-9d1c...", // UUID — permanent identity "outputs": { ... }, "resources": [{ "type": "google_compute_instance", "name": "web_server", "instances": [{ "attributes": { "id": "projects/.../instances/web", "machine_type": "e2-medium", "network_ip": "10.0.0.5" // ALL cloud-returned attributes stored here } }] }] }
backend.tf — Remote State on GCS (Production Required)
terraform { backend "gcs" { bucket = "my-org-tf-state-prod" prefix = "infra/gke-cluster" encryption_key = var.kms_key_id // CMEK } } # State locking: GCS creates .tflock object atomically # if-generation-match:0 prevents two concurrent writes # No DynamoDB needed — GCS handles it natively! # Migrate local → GCS: terraform init -migrate-state # Env-specific: terraform init -backend-config=prod.tfbackend
◈ State Locking — Two Engineers Fight Over State
❌ WITHOUT LOCKING
Amir runs apply → reads state v12
Priya runs apply → reads state v12
Amir writes state v13 ✓
Priya writes state v13 — OVERWRITES AMIR!
State v13 = corrupted. Infra broken.
✅ WITH GCS LOCKING
Amir runs apply → 🔒 LOCK acquired
Priya runs apply → BLOCKED — lock held
Amir completes → 🔓 LOCK released
Priya's turn → reads fresh state
State consistent. Everyone safe.
NEVER commit .tfstate to git. Add *.tfstate and *.tfstate.backup to .gitignore. State contains secrets in plain text.
Enable GCS object versioning on your state bucket — gives you a time machine. terraform force-unlock LOCK_ID when an apply crashes mid-run.

Complete Module Pattern
# modules/vpc/variables.tf — the module's API variable "network_name" { type = string } variable "subnets" { type = map(object({ cidr = string })) } # modules/vpc/main.tf resource "google_compute_network" "vpc" { name = var.network_name auto_create_subnetworks = false } resource "google_compute_subnetwork" "subnets" { for_each = var.subnets name = "${var.network_name}-${each.key}" network = google_compute_network.vpc.id ip_cidr_range = each.value.cidr } # modules/vpc/outputs.tf output "network_id" { value = google_compute_network.vpc.id } output "subnet_ids" { value = { for k, v in google_compute_subnetwork.subnets : k => v.id } } # root/main.tf — calling the module module "network" { source = "./modules/vpc" # local path network_name = "prod-vpc" subnets = { "app" = { cidr = "10.0.1.0/24" } "data" = { cidr = "10.0.2.0/24" } } } module "gke" { source = "./modules/gke" network_id = module.network.network_id ← output → input }

Module source types (3 options):

  • 📁 Local: source = "./modules/vpc"
  • 🌐 Registry: source = "terraform-google-modules/network/google"
  • 🔗 Git: source = "git::https://github.com/org/repo.git//modules/vpc?ref=v1.0"

Rules: Never put backend {} inside a module. Never put provider {} inside a module. ALWAYS pin registry modules with version = "~> X.0"

Level 2 · Dynamic Resources

count vs for_each — Clone Army vs Named Squad

⚠️ count = CLONE ARMY
Soldiers: [0], [1], [2]
Remove [1] → [2] becomes [1]
Terraform sees CHANGE → may destroy and recreate [1]
Index shift = unexpected destruction!
[0]
[1] 💥
[2]→[1] ?
✅ for_each = NAMED SQUAD
Soldiers: ["alpha"], ["bravo"], ["charlie"]
Remove "bravo" → alpha and charlie unchanged
Their names never change
Keys are stable identities!
alpha ✓
bravo ✗
charlie ✓
All Loop Patterns
# ── count: N identical copies ───────────────────────────── resource "google_compute_instance" "web" { count = 3 name = "web-${count.index}" # web-0, web-1, web-2 } # State: google_compute_instance.web[0], web[1], web[2] # ── count: conditional resource ────────────────────────── resource "google_compute_instance" "bastion" { count = var.enable_bastion ? 1 : 0 # exists or doesn't } # ── for_each: set of strings ────────────────────────────── resource "google_storage_bucket" "buckets" { for_each = toset(["logs", "assets", "backups"]) name = "my-org-${each.key}" } # State: google_storage_bucket.buckets["logs"] etc. # ── for_each: map of objects (most powerful) ───────────── resource "google_container_node_pool" "pools" { for_each = { "general" = { machine_type="e2-standard-4", min=1, max=5 } "spot" = { machine_type="e2-standard-8", min=0, max=10 } } name = each.key machine_type = each.value.machine_type } # ── dynamic block: for_each INSIDE resource ────────────── resource "google_compute_firewall" "web" { dynamic "allow" { for_each = [80, 443, 8080] content { protocol = "tcp" ports = [tostring(allow.value)] } } } # ── for expression: transform collections ───────────────── locals { subnet_map = { for s in var.subnets : s.name => s.cidr } prod_pools = { for k,v in var.pools : k => v if !v.spot } } # ── flatten + for: cross-product (SA × role pairs) ──────── locals { sa_roles = { for p in flatten([ for sa in var.sas : [for r in var.roles : {key="${sa}-${r}",sa=sa,role=r}] ]) : p.key => p } }

Golden Rule: default to for_each. Use count ONLY for: (a) N identical clones, (b) conditional 0/1.

Pitfalls:

  • for_each needs a set or map, not a list → wrap with toset()
  • Keys must be known at plan time — can't use auto-generated IDs
  • Can't use BOTH count and for_each on same resource
Level 3 · Lifecycle

Lifecycle — Control Resource Behaviour

All lifecycle options
resource "google_sql_database_instance" "db" { lifecycle { prevent_destroy = true # ↑ terraform destroy FAILS if this is true # Use on: databases, state buckets, prod clusters create_before_destroy = true # ↑ when forced replace: create new FIRST, then destroy old # Zero-downtime replacements ignore_changes = [settings[0].disk_size] # ↑ ignore drift on this attribute (GCP auto-grows disk) # Use sparingly — can hide real problems replace_triggered_by = [google_compute_image.new] # ↑ force replace THIS resource when another changes } }
prevent_destroy is your safety net for production databases. But note: terraform destroy on the MODULE containing the resource still works — prevent_destroy only blocks the resource's direct deletion.

Jenkinsfile (Declarative)
pipeline { agent { label 'terraform' } environment { GOOGLE_CREDENTIALS = credentials('gcp-tf-sa-key') TF_VAR_project_id = 'my-project-id' } stages { stage('Init') { steps { sh 'terraform init -input=false' } } stage('Validate') { steps { sh 'terraform validate' } } stage('Plan') { steps { sh 'terraform plan -out=tfplan' } } stage('Approval') { when { branch 'main' } steps { input message: 'Review plan. Apply?', submitter: 'devops-leads' } } stage('Apply') { when { branch 'main' } steps { sh 'terraform apply -auto-approve tfplan' } } } }
Workload Identity Federation — No SA Key Files!
# WIF = GitHub's OIDC token exchanged for short-lived GCP creds # No JSON key files. No rotation. No leaks. Modern standard. # Step 1: Create WIF pool + provider gcloud iam workload-identity-pools create "github-pool" \ --location=global gcloud iam workload-identity-pools providers create-oidc "github" \ --workload-identity-pool="github-pool" \ --issuer-uri="https://token.actions.githubusercontent.com" \ --attribute-mapping="google.subject=assertion.sub,attribute.repository=assertion.repository" # Step 2: Bind SA to WIF gcloud iam service-accounts add-iam-policy-binding \ [email protected] \ --role="roles/iam.workloadIdentityUser" \ --member="principalSet://iam.googleapis.com/.../attribute.repository/org/repo" # Plan SA: only refs/heads/* (any branch) # Apply SA: only refs/heads/main (main branch ONLY!)
NEVER run terraform apply on a pull_request event. Only on push to main. Apply on PR = infra changes before code review. The car rolls off the line before the inspector signs it.
Level 3 · Security

Security — Three Castle Walls

🏰 Medieval Castle Outer wall → inner wall → vault. Each wall stops a different attacker.
=
🔐 Terraform Security Code scanning → IAM → state encryption. Each layer stops different threats.
🔍
SAST: tfsec + checkov
"Spell-check for security"
Scan .tf files on every PR. Catch open firewalls, public buckets, missing encryption BEFORE deploy. Block PR on HIGH severity findings.
WALL 1: CODE
🔑
Secrets: Never in .tf
"Don't write passwords on the blueprint"
Use Secret Manager refs. Terraform manages the secret RESOURCE not the VALUE. sensitive=true hides from terminal, NOT from state.
WALL 1: CODE
🏦
State Encryption (CMEK)
"Lock the vault with your own key"
GCS bucket + Cloud KMS CMEK + object versioning. iam_binding (authoritative) on bucket. Enable Data Access audit logs.
WALL 2: STATE
👤
Least Privilege SA
"Give the locksmith only the keys he needs"
NEVER roles/owner or roles/editor. Enumerate specific roles: compute.admin, container.admin. Plan SA = viewer only. Apply SA = specific admins.
WALL 3: IAM
🎫
WIF — No Key Files
"Use your badge, not a spare key"
Workload Identity Federation replaces SA JSON keys. GitHub OIDC token → short-lived GCP credential. No rotation. No leaks. 2025 standard.
WALL 3: IAM
📜
OPA / Conftest
"Policy officer reviews the plan before execution"
Policy as code on the plan JSON. Catches logic flaws SAST misses (variable that conditionally makes bucket public). Runs in CI after plan.

Insecure vs Secure Patterns

-target — Apply one resource
